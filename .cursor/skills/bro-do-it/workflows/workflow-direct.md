# Режим `direct` (контекст без готового плана)

Используй, когда пользователь просит реализовать задачу **без** ссылки на готовый план, `.plan.md` или `todos` из плана (см. правила маршрутизации в `rules/do-it.mdc`).

Контракт субагентов, запрет собственных правок оркестратора и ревью-цикл: [SKILL.md](../SKILL.md) (разделы 3–4).

Параметры `Task` для ролей — таблица в [SKILL.md](../SKILL.md), «Механика вызова».

## Основные правила

1. СНАЧАЛА определи текущую задачу, рамки и критерии приемки из контекста разговора и кода.
2. Если задача, рамки или критерии приемки **неясны**, **остановись** и **запроси уточнение** у пользователя.
3. НЕ передавай весь контекст в субагента — только необходимое для текущей роли.
4. Если задачу можно безопасно разбить на независимые части без пересечения по файлам и зависимостям, МОЖНО запускать несколько /developer (`subagent_type=developer`, `model=composer-2.5`) параллельно.
5. НЕ создавай отдельный план, если пользователь этого не просил.
6. Оркестратор **не** вносит собственных правок в репозиторий — см. [SKILL.md](../SKILL.md), раздел 3.

## Цикл оркестрации

1. Определи задачу, рамки и критерии приемки из запроса и доступного контекста.
2. При недостатке данных для безопасного старта — стоп и уточняющий вопрос пользователю.
3. Запусти /developer (`subagent_type=developer`, `model=composer-2.5`) с узким промптом.
4. При нескольких независимых подзадачах — несколько /developer (`subagent_type=developer`, `model=composer-2.5`) параллельно только без пересечений по файлам и зависимостям.
5. После реализации текущего набора изменений запусти /reviewer (`subagent_type=reviewer`, `model=gpt-5.3-codex`).
6. Далее следуй ревью-циклу в [SKILL.md](../SKILL.md), раздел 4.

# Режим `plan` (готовый план / `.plan.md` / `todos`)

Используй, когда пользователь явно исполняет **уже существующий** план (см. `rules/build-plan.mdc`). **Не** составляй и **не** переписывай план в этом режиме.

Если подходящего артефакта плана **нет**, **остановись** и сообщи пользователю.

Контракт субагентов, единственное исключение на собственные правки оркестратора и ревью-цикл: [SKILL.md](../SKILL.md) (разделы 3–4).

Параметры `Task` для ролей — таблица в [SKILL.md](../SKILL.md), «Механика вызова».

## Основные правила

1. СНАЧАЛА **прочитай** файл плана, чтобы понять, на каком этапе остановились, и определить следующий шаг.
2. НЕ передавай весь план в субагента — только необходимое для текущей задачи/шага.
3. Если исполняется Cursor `.plan.md` с frontmatter `todos`, **поддерживай** статусы `todos` **только** в этом `.plan.md` (см. ниже). Это **единственный** вид собственных правок оркестратора в этом режиме.
4. Оркестратор **не** меняет исходный код, тесты, конфиги, миграции и любые иные артефакты — см. [SKILL.md](../SKILL.md), раздел 3.

## Цикл оркестрации

1. Определи, какую задачу делать **следующей** по плану, списку шагов и `todos`.
2. Если это Cursor `.plan.md` с `todos`, переведи текущий `todo` в `in_progress` до запуска /developer (`subagent_type=developer`, `model=composer-2.5`).
3. Запусти /developer (`subagent_type=developer`, `model=composer-2.5`) с узким промптом.
4. Если задачи можно выполнять параллельно, запускай **несколько** /developer (`subagent_type=developer`, `model=composer-2.5`) под **отдельную** задачу (без конфликтов по артефактам).
5. После завершения шага отметь соответствующий `todo` как `completed` и переходи к следующему шагу.
6. **После реализации всех шагов** плана запусти /reviewer (`subagent_type=reviewer`, `model=gpt-5.3-codex`) на текущий набор правок, передав **весь план** и **выполненные изменения** (в промпте субагента, без ожидания что он сам прочитает файлы проекта).
7. Далее следуй ревью-циклу в [SKILL.md](../SKILL.md), раздел 4.

## Синхронизация `todos` в `.plan.md`

Обязательно **только** для Cursor `.plan.md` с frontmatter `todos`. Оркестратор обновляет **только** этот файл `.plan.md` — не другие плановые или проектные артефакты.

1. До запуска /developer (`subagent_type=developer`, `model=composer-2.5`): текущий `todo` → `in_progress`.
2. На время реализации сохраняй `in_progress`.
3. После решения задачи: соответствующий `todo` → `completed`.
4. При остановке из‑за блокера, нехватки данных или после второго ревью с нерешёнными находками **`todo` не помечай** как `completed`.
5. Отмена (`cancelled`) — только при **явном** подтверждении пользователем смены рамок.
6. При обновлении frontmatter сохраняй остальные поля `.plan.md` без самовольных изменений.

## Эскалация

При отклонении от процесса или затруднениях **остановись** и сообщи пользователю.


# plan

Реализация задачи когда **существует** файл/артефакт план.

## Основные правила

1. СНАЧАЛА **прочитай** файл плана, чтобы понять, на каком этапе остановились, и определить следующий шаг.
2. Если задача, рамки или критерии приемки **неясны**, **остановись** и **запроси уточнение** у пользователя.

## Контрольные точки `[CHECKPOINT]`

CHECKPOINT — это не шаг реализации, а обязательная остановка оркестратора.

Developer **НИКОГДА** не должен получать:
- шаг `[CHECKPOINT]`
- шаги после ближайшего неапрувленного CHECKPOINT

Перед **КАЖДЫМ** новым батчем developer **ОБЯЗАТЕЛЬНО**:

1. Найди первый незавершённый шаг или todo.
2. Если это CHECKPOINT:
   - **не** вызывай developer и **не** меняй исходный код
   - кратко отчитайся, что сделано в предыдущем батче
   - перечисли, что нужно проверить человеку
   - явно укажи, какой следующий этап будет разблокирован после апрува
   - остановись и дождись явного подтверждения пользователя именно на этот CHECKPOINT
3. Только после явного подтверждения пользователя отметь checkpoint как `completed`.
4. Затем снова определи первый незавершённый шаг.
5. Если первый незавершённый шаг не CHECKPOINT, собери батч только из подряд идущих шагов реализации до ближайшего CHECKPOINT.
6. После завершения батча снова вернись к пункту 1.

Команды вида `/bro-do-it`, «продолжай», «делай дальше», «реализуй план» **не считаются** апрувом CHECKPOINT без явного запроса и явного ответа пользователя на конкретную контрольную точку.

Шаги, меняющие тестовые файлы, выполняй **только** после прохождения `[CHECKPOINT: апрув перед тестами]`.

Шаблон ответа на CHECKPOINT:
```
Дошли до `<название checkpoint>`.

Сделано:
- ...
- ...

Что нужно проверить:
- ...
- ...

После вашего апрува будет разблокирован этап:
- `<название следующего этапа>`

До вашего подтверждения я НЕ буду вызывать developer на следующие шаги.

Если ок, подтвердите переход к `<следующий этап>`.
```

## Формирование списка шагов

Прочитай план и на его основе сформируй список шагов.
Если в плане выделен список `todos`, используй его в качестве шагов.

Каждый шаг должен содержать:
- где меняем — **полный путь к файлу** от корня репозитория; имя класса, функции или метода — дополнение к пути, не замена
- что меняем
- зачем это делаем

Пример шаблона шагов:
```
## Шаг 1
- Где меняем: `src/orders/OrderService.ts` (класс `OrderService`, метод `recalculateTotal`)
- Что меняем: ...
- Зачем это делаем: ...

## Шаг 2
- Где меняем: `src/orders/OrderRepository.ts` (класс `OrderRepository`)
- Что меняем: ...
- Зачем это делаем: ...
```

## Синхронизация `todos`

Если в плане используется список `todos` со статусами, то **ОБЯЗАТЕЛЬНО** меняй статус шага:
* при начале [реализации](../SKILL.md#2-реализация) отмечай шаг "in_progress"
* при завершении реализации отмечай шаг "completed"
* для шага `[CHECKPOINT]` — `completed` только после явного апрува пользователя

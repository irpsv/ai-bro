# Правила структуры stage implementation plan

Цель: stage-план должен быть читаем **без интерпретаций** — исполнитель и ревьюер понимают границы, конечное состояние и проверку результата без догадок.

## Порядок секций

Stage-план **ДОЛЖЕН** следовать цепочке:

`зачем -> границы -> источник истины -> конечное состояние -> шаги -> проверка результата`

Не переходи к шагам, пока не зафиксированы границы этапа и конечное состояние.

## Обязательные секции

### 1. Заголовок и вводный абзац

- Заголовок: `# Этап реализации плана <имя этапа>`.
- Вводный абзац **ДОЛЖЕН** содержать:
  - что делает этап;
  - что этап **явно не делает** (non-goals), если это неочевидно из `plan.md`.

### 2. Что будет готово после этапа

- Формулируй в терминах **конечных артефактов и контрактов**, а не действий.
- Перечисляй файлы, сущности, интерфейсы, записи, тесты — всё, что должно существовать после этапа.
- Если вводятся закрытые перечни (enum, коды, матрицы), фиксируй их здесь или в шагах с формулами `ровно`, `полный перечень`.
- Итог этапа **ДОЛЖЕН** быть читаем без интерпретаций: по документу видно, что именно появится в системе.

### 3. Шаги реализации

- Один заголовок `## Шаг N` на каждое логическое изменение.
- Формат шага — по [step-rules.md](./step-rules.md).
- Шаги — средство достижения заранее зафиксированного результата, а не единственное содержание документа.

### 4. Критерии приемки

- Нумерованный список проверяемых утверждений.
- Каждый пункт **ДОЛЖЕН** быть измерим или наблюдаем: «содержит ровно N», «возвращает ровно …», «созданы отсутствующие записи», «повторный запуск не создаёт дублей».
- Критерии **ДОЛЖНЫ** отражать ключевые deliverables из раздела «Что будет готово после этапа» и шагов.
- Избегай субъективных формулировок («работает хорошо», «покрыто тестами» без конкретики).

## Опциональные секции

Добавляй, если без них возможна двусмысленность:

### Источник истины для этапа

- Конкретные пути к файлам, документам или поведению, на которые опирается план.
- Правило разрешения конфликтов: например, «если план расходится с кодом, для этого этапа источником истины считается код».

### Нормализация терминов

- Определения доменных групп, сокращений, неочевидных имён.
- Привязка к реальным типам, enum или API, если это снимает двусмысленность.

## Инварианты качества

1. **Без интерпретаций** — исполнитель не должен угадывать scope, состав артефактов или правила проверки.
2. **Границы на двух уровнях** — на уровне этапа (вводный абзац) и на уровне неоднозначных шагов (подраздел «Что не делаем»).
3. **Дублирование матриц допустимо** — одна и та же бизнес-матрица может повторяться в deliverables, шагах и критериях приемки, если это повышает однозначность.
4. **Закрытые перечни** — где важна полнота, используй `ровно`, `полный перечень`, `не добавлять другие варианты`.
5. **Контракты без потребителей** — если этап готовит только контракты для следующих этапов, явно укажи, что runtime-потребители в этот этап не входят.
6. **Согласованность с plan.md** — stage-план не противоречит целям, рамкам и описанию этапа в общем плане; при расхождении с кодом — сначала согласование с человеком.

## Frontmatter stage.md

- `path-to-plan` — ссылка на родительский `plan.md`.
- `steps` — список шагов с `name` и `status`; имена **ДОЛЖНЫ** совпадать с заголовками `## Шаг N` в теле.
- При создании stage-артефакта все шаги в frontmatter — `wait`.

# Thinker prompt template (bro-remember-it)

Use when dispatching thinker for memory-bank placement recommendations.

## Tool

- Use tool `Task` (`subagent_type=thinker`)
- Agent definition: `.cursor/agents/thinker.md` (`readonly: true`)
- **Do not** use fast mode for this task

## Prompt template

```
Ты thinker-субагент для READONLY анализа: куда и что положить в project memory-bank после выполненных изменений.

## READONLY

- Разрешено: читать репозиторий, memory-bank, diff, план, конфиги.
- Запрещено: править любые файлы, создавать коммиты, писать готовые патчи/вставки в bank (только рекомендации родителю).

## Задача

1. Прочитай текущий memory-bank по пути <путь до memory-bank>. Сам оцени структуру, индекс, существующие документы, дубликаты и конфликты.
2. Сопоставь с переданными изменениями (ревизии/diff) и контекстом «что и зачем делалось».
3. Верни список рекомендаций: **КУДА**, **ЧТО**, **ПОЧЕМУ** — только для знаний, которые **можно переиспользовать** и которые **полезны в будущем** (продукт, поведение, инварианты, контракты, сценарии, архитектурные решения с продуктовым смыслом).

**Не рекомендуй**: детали кода, рефакторинг ради стиля, процессные уроки, одноразовые костыли, то, что уже очевидно из кода без потери смысла для будущих задач.

Если в memory-bank ничего не нужно обновлять — так и скажи, с кратким обоснованием.

## Вход от родителя

### Memory-bank

<корень memory-bank, полный путь>
<опционально: индекс/README>

### Что и зачем делалось

<путь до плана ИЛИ выжимка из сессии>

### Изменения (ревизии)

<список номеров коммитов и ревизий>

## Формат ответа

Строго на русском. Заголовки `###` с **точными** названиями, в указанном порядке.

### Обзор memory-bank

Кратко: как устроен bank, какие разделы релевантны этим изменениям (пути).

### Рекомендации

Для каждой рекомендации — подзаголовок `#### Рекомендация N` и поля:

- **действие**: создать | дополнить | заменить раздел | пометить устаревшим | не трогать
- **куда**: полный путь `...` (раздел/якорь, если применимо)
- **что**: суть знания для bank (кратко, без лишней реализации)
- **почему**: связь с изменением (файл, коммит, пункт «что и зачем»)
- **приоритет**: must | should | nice
- **дубликат/конфликт**: если есть — что уже в bank и что делать

### Отклонено от фиксации

Что из изменений **намеренно не** кладём в bank и почему (краткий список).

## Правила качества

- Доказательства важнее догадок; при низкой уверенности снижай приоритет и помечай.
- Не смешивай несколько разных знаний в одну рекомендацию, если у них разные target.
- Предпочитай **дополнить** существующий документ, а не плодить новые файлы, если смысл тот же.
- План vs факт: в bank фиксируй **фактическое** состояние продукта после изменений.
```

# Bro Give Me Plan

Используй этот скилл для составления плана реализации (implementation plan).
**НЕ ПИШИ** сам код, только составляй план.

## ОБЩИЕ ПРАВИЛА

1. **ОБЯЗАТЕЛЬНО** спрашивай по одному вопросу, НЕ спрашивай все разом!

2. **ОБЯЗАТЕЛЬНО** соблюдай и **НЕ нарушай** порядок разделов плана!

3. **СНАЧАЛА СОБЕРИ САМОСТОЯТЕЛЬНО** необходимую информацию:
   - Не спрашивай то, что можно установить с высокой уверенностью.
   - Помечай выведенное как выведенное, пока пользователь его не подтвердит.
   - Для шагов реализации заранее найди в кодовой базе **проверенные пути к файлам** (grep, read, поиск по проекту) — не указывай путь «на глаз».

4. план **ДОЛЖЕН** быть однозначным, без наличия вариаций и вопросов в нём.

5. **ОБЯЗАТЕЛЬНО** нужно выяснять все детали и подробности **ДО** создания плана, чтобы в самом плане была только конкретная информация.

## Порядок работы

1. **СНАЧАЛА** собери всю информацию и необходимый контекст
2. **ПРОАНАЛИЗИРУЙ** информацию и определи чего тебе не хватает для заполнения всех разделов плана
3. **ЗАСПРОСИ** у человека необходимую информацию. **ОБЯЗАТЕЛЬНО** соблюдай [правила диалога](#правила-диалога)
4. **СОХРАНИ** план через `CreatePlan` согласно [сохранению плана](#сохранение-плана-createplan)

## Правила диалога

Если нужно задать вопросы пользователю **ОБЯЗАТЕЛЬНО** задавай 1 вопрос за 1 раз.
Если есть варианты решений **ПРЕДЛОЖИ** их пользователю.
**ОБЯЗАТЕЛЬНО** нумеруй предлагаемые варианты ответов: 1) 2) 3) ...

Используй tool `AskQuestion` (или аналог) если выполняются ВСЕ условия:
   - вопрос до 500 символов
   - варианты ответов до 200 символов
   - не требуется развернутый ответ
   - не требуется дополнительый контекст

## Формат плана

### Разделы плана

Ровно в таком порядке:
1. задача и контекст
2. целевой результат
3. рамки задачи
4. шаги реализации
5. критерии приемки

#### 1. Задача и контекст

Цель: сдалть прозрачным контекст и понимание для решения задачи.

Нужно покрыть:
- какую проблему решаем
- почему именно сейчас
- кто затронут
- как выглядит текущее состояние
- какие есть подтверждения проблемы, если они есть

Спрашивай только если неясно:
- кто страдает
- почему важен тайминг
- что именно не так в текущем поведении

Критерии выхода:
- проблема сформулирована конкретно
- затронутый актор назван
- срочность объяснена или помечена как неизвестная

#### 2. Целевой результат

Цель: обозначить целевой результат.

Нужно покрыть:
- идеальный конечный результат
- конкретные пользовательские или системные сценарии
- что должно стать возможным после изменения

Критерии выхода:
- зафиксирован хотя бы один конкретный сценарий успеха
- успех описан через поведение, а не только через намерение

#### 3. Рамки задачи

Цель: явно зафиксировать границы.

Нужно зафиксировать дословно:
- что мы явно не делаем в рамках этой задачи

Также нужно покрыть:
- что входит в рамки задачи
- что намеренно оставляем за рамками

Критерии выхода:
- видно и что входит в задачу, и что не входит

#### 4. Шаги реализации

Цель: детально зафиксировать планируемые изменения

Нужно покрыть:
- все действия необходимые при выполнении плана

Каждый шаг заполняется в формате:
- где меняем — **полный путь к файлу** от корня репозитория; имя класса, функции или метода — дополнение к пути, не замена
- что меняем
- зачем это делаем

Перед сохранением плана для каждого шага в «где меняем» должен быть **проверенный** путь к файлу из исследования кодовой базы (см. [п. 3 общих правил](#общие-правила)).

Пример шаблона шагов:
```
## Шаг 1
- Где меняем: `src/orders/OrderService.ts` (класс `OrderService`, метод `recalculateTotal`)
- Что меняем: ...
- Зачем это делаем: ...

## Шаг 2
- Где меняем: `src/orders/OrderRepository.ts` (класс `OrderRepository`)
- Что меняем: ...
- Зачем это делаем: ...
```

Правила формирования шагов:
- В поле **«Где меняем»** указывай **полный путь к файлу** от корня репозитория (например `src/api/handlers/user.ts`). Имя класса, функции или метода — **дополнение** к пути, не замена. Путь должен быть найден при исследовании кодовой базы до сохранения плана; не указывай путь «на глаз». Если затрагивается несколько файлов — отдельный шаг на файл или перечисление путей в одном шаге списком.
- **УПОРЯДОЧИВАЙ** шаги в практической последовательности выполнения.
- **РАЗБИВАЙ** крупные изменения на отдельные шаги.
- **НЕ УКАЗЫВАЙ** области, которые вы не можете обосновать (в том числе путь к файлу, если он не найден).
- **НЕ ОБЪЕДИНЯЙ** несвязанные изменения в один пункт.

#### 5. Критерии приемки и проверка

Цель: зафиксировать измеримое завершение работы.

Нужно зафиксировать дословно:
- измеримые условия приемки

Также нужно покрыть:
- как будет проверяться результат
- какие сценарии должны проходить

Критерии выхода:
- критерии приемки можно проверить
- способ проверки назван или помечен как неизвестный

### Сохранение плана (CreatePlan)

План **сохраняй только** через tool `CreatePlan`. Frontmatter файла `.plan.md` (`name`, `overview`, `todos`, `isProject`) формирует Cursor из аргументов вызова — **не дублируй** его в markdown.

#### Аргументы `CreatePlan`

| Аргумент | Назначение |
|----------|------------|
| `name` | Короткое имя плана |
| `overview` | 1–2 предложения: суть и границы |
| `todos` | Список задач исполнения (см. ниже) |
| `plan` | Только markdown-тело: заголовок и разделы 1–5 |

#### `todos` (frontmatter)

**ОБЯЗАТЕЛЬНО** для каждого шага из раздела «4. шаги реализации» — отдельный элемент в аргументе `todos`:

- `id` — стабильный kebab-case (латиница, цифры, дефисы)
- `content` — короткое проверяемое действие
- `status` — `pending` при создании плана

При **обновлении** существующего `.plan.md`: сначала прочитай файл, **не переписывай** `todos` вслепую — сохрани существующие `id`, актуализируй `content`/`status` под новый план.

#### `plan` (markdown-тело)

Структура аргумента `plan`:

1. Один заголовок `# …`
2. Разделы **ровно** в порядке из [формата плана](#разделы-плана): `## 1. …` … `## 5. …`
3. Внутри «4. шаги реализации» — подшаги `## Шаг N` по шаблону выше

**ЗАПРЕЩЕНО** в аргументе `plan`:

- `todos:` и любые YAML-списки задач
- блоки `---` … `---` с метаданными или todos
- повтор `name` / `overview` из frontmatter

#### Проверка перед вызовом `CreatePlan`

- Каждый `## Шаг N` из тела имеет соответствующий элемент в `todos`
- В тексте `plan` нет строки `todos:` и нет YAML frontmatter

# Bro Give Me Plan

Используй этот скилл для составления плана реализации (implementation plan).
**НЕ ПИШИ** сам код, только составляй план.

## ОБЩИЕ ПРАВИЛА

1. **ОБЯЗАТЕЛЬНО** спрашивай по одному вопросу, НЕ спрашивай все разом!

2. **ОБЯЗАТЕЛЬНО** соблюдай и **НЕ нарушай** порядок разделов плана!

3. **СНАЧАЛА СОБЕРИ САМОСТОЯТЕЛЬНО** необходимую информацию для §1–§3:
   - Не спрашивай то, что можно установить с высокой уверенностью.
   - Помечай выведенное как выведенное, пока пользователь его не подтвердит.

4. План **ДОЛЖЕН** быть однозначным, без наличия вариаций и вопросов в нём.

5. **ОБЯЗАТЕЛЬНО** выясни все детали для §1–§3 **ДО** делегирования [thinker](#thinker), чтобы в финальном плане была только конкретная информация.

## STOP

**ЗАПРЕЩЕНО** самостоятельно писать шаги плана и критерии приемки, ОБЯЗАТЕЛЬНО делегируй эту работу субагенту [thinker](#thinker).

## Субагенты

### thinker

READONLY субагент для формирования шагов исполнения и критериев приемки **§4–§5** (до `CreatePlan`). **ОБЯЗАТЕЛЬНО** промпт [plan-author-prompt.md](./subagents/plan-author-prompt.md). На вход — пакет §1–§3; **не** полная история чата.

## Анти-паттерны (нарушение = эскалация)

| Нарушение | Почему плохо |
|-----------|----------------|
| `CreatePlan` до проверки черновика §4–§5 | План без проверки scope и путей |
| Оркестратор сам пишет §4–§5 без thinker | Обход делегирования исполнимой части |
| Передать thinker итоговый план или результат Quality Control | Лишний цикл и шум |
| Передать thinker полную историю чата | Перерасход контекста |
| Больше 2 вызовов thinker на одну задачу | Зависание сессии |
| Бесконечные уточнения у человека без лимита | Зависание сессии |

## Порядок работы

1. **СОБЕРИ** информацию для §1–§3: код, контекст, при необходимости — [правила диалога](#правила-диалога). **ПРОЧИТАЙ** [plan-format.md](./references/plan-format.md).
2. **ДЕЛЕГИРУЙ** thinker: пакет §1–§3 по [plan-author-prompt.md](./subagents/plan-author-prompt.md).
3. При `needs_clarification` от thinker — **ЗАДАЙ** вопросы человеку, строго согласно [правилам диалога](#правила-диалога). Далее повторный вызов thinker с обновлённым пакетом (не более **2** вызовов thinker на задачу).
4. **ПРОВЕРЬ** полученный ответ от thinker после шага 2 по согласно правилам в [quality control](#quality-control); исправь при необходимости.
5. **СОХРАНИ** план: **ОБЯЗАТЕЛЬНО** прочитай [create-plan.md](./references/create-plan.md), сверь тело с [plan-format.md](./references/plan-format.md), в качестве §4–§5 используй результат работы thinker и вызови `CreatePlan` строго по `create-plan.md`.

## Правила диалога

Если нужно задать вопросы пользователю **ОБЯЗАТЕЛЬНО** задавай 1 вопрос за 1 раз.
Если есть варианты решений **ПРЕДЛОЖИ** их пользователю.
**ОБЯЗАТЕЛЬНО** нумеруй предлагаемые варианты ответов: 1) 2) 3) ...

Используй tool `AskQuestion` (или аналог) если выполняются ВСЕ условия:
   - вопрос до 500 символов
   - варианты ответов до 200 символов
   - не требуется развернутый ответ
   - не требуется дополнительый контекст

## Формат плана

Полные требования к каждому разделу — в [plan-format.md](./references/plan-format.md). Ровно **5 разделов**, в таком порядке:

| # | Раздел | Кто готовит |
|---|--------|-------------|
| 1 | задача и контекст | оркестратор |
| 2 | целевой результат | оркестратор |
| 3 | рамки задачи | оркестратор |
| 4 | шаги реализации | thinker → Quality Control оркестратора |
| 5 | критерии приемки | thinker → Quality Control оркестратора |


## Quality Control

Перед `CreatePlan` проверь черновик плана:

1. **Scope:** «что входит / что не делаем» (§3) покрыто шагами §4 и не нарушено.
2. **Пути:** каждый путь в §4 существует в репозитории (grep/read).
3. **Согласованность:** §3 ↔ §4 ↔ §5.

Правки после Quality Control вносит **только оркестратор**. Thinker повторно **не** вызывай.

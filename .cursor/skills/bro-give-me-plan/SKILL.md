# Bro Give Me Plan

Используй этот скилл для составления плана реализации (implementation plan).
**НЕ ПИШИ** сам код, только составляй план.

## ОБЩИЕ ПРАВИЛА

1. **ОБЯЗАТЕЛЬНО** спрашивай по одному вопросу, НЕ спрашивай все разом!

2. **ОБЯЗАТЕЛЬНО** соблюдай и **НЕ нарушай** порядок разделов плана!

3. **СНАЧАЛА СОБЕРИ САМОСТОЯТЕЛЬНО** необходимую информацию:
   - Не спрашивай то, что можно установить с высокой уверенностью.
   - Помечай выведенное как выведенное, пока пользователь его не подтвердит.
   - Для шагов реализации заранее найди в кодовой базе **проверенные пути к файлам** (grep, read, поиск по проекту) — не указывай путь «на глаз».

4. план **ДОЛЖЕН** быть однозначным, без наличия вариаций и вопросов в нём.

5. **ОБЯЗАТЕЛЬНО** нужно выяснять все детали и подробности **ДО** создания плана, чтобы в самом плане была только конкретная информация.

6. **ОБЯЗАТЕЛЬНО** после сохранения итогового плана делегируй критику субагенту [thinker](#thinker) — см. [порядок работы](#порядок-работы) и [субагентов](#субагенты).


## Субагенты

### thinker

READONLY критика **итогового** плана (файл `.plan.md` после `CreatePlan`). **ОБЯЗАТЕЛЬНО** промпт [plan-critic-prompt.md](./subagents/plan-critic-prompt.md). На вход thinker — **только путь** к файлу плана.

## Анти-паттерны (нарушение = эскалация)

| Нарушение | Почему плохо |
|-----------|----------------|
| Не вызвать thinker после сохранения итогового плана | Нет adversarial review |
| Критиковать неполный/промежуточный план | Ложные ok или бессмысленные findings |
| Оркестратор сам «критикует» вместо thinker | Тот же blind spot |
| Передать thinker тело плана вместо пути / всю историю чата | Дублирование, шум, рассинхрон с файлом |
| Дублировать критерии критика в `references/` | Два источника правды |
| Бесконечные циклы критика | Зависание сессии |

## Порядок работы

1. **СНАЧАЛА** собери всю информацию и необходимый контекст
2. **ПРОЧИТАЙ** [plan-format.md](./references/plan-format.md)
3. **ПРОАНАЛИЗИРУЙ** информацию и определи, чего не хватает для заполнения всех разделов плана
4. **ЗАСПРОСИ** у человека необходимую информацию. **ОБЯЗАТЕЛЬНО** соблюдай [правила диалога](#правила-диалога)
5. **СОХРАНИ** план: **ОБЯЗАТЕЛЬНО** прочитай [create-plan.md](./references/create-plan.md), сверь тело с [plan-format.md](./references/plan-format.md) и вызови `CreatePlan` строго по `create-plan.md`
6. **ЗАПУСТИ** субагента [thinker](#thinker) для проверки плана — [plan-critic-prompt.md](./subagents/plan-critic-prompt.md)
7. **ВНЕСИ** исправления после шага 6 **ТОЛЬКО ЕСЛИ** они были

## Правила диалога

Если нужно задать вопросы пользователю **ОБЯЗАТЕЛЬНО** задавай 1 вопрос за 1 раз.
Если есть варианты решений **ПРЕДЛОЖИ** их пользователю.
**ОБЯЗАТЕЛЬНО** нумеруй предлагаемые варианты ответов: 1) 2) 3) ...

Используй tool `AskQuestion` (или аналог) если выполняются ВСЕ условия:
   - вопрос до 500 символов
   - варианты ответов до 200 символов
   - не требуется развернутый ответ
   - не требуется дополнительый контекст

## Формат плана

Полные требования к каждому разделу — в [plan-format.md](./references/plan-format.md). Ровно **5 разделов**, в таком порядке:

| # | Раздел | Зачем |
|---|--------|--------|
| 1 | задача и контекст | проблема, актор, текущее состояние, срочность |
| 2 | целевой результат | сценарии успеха через поведение |
| 3 | рамки задачи | что входит и что явно не делаем |
| 4 | шаги реализации | где / что / зачем, с проверенными путями к файлам |
| 5 | критерии приемки | измеримые условия и способ проверки |

# Bro Give Me Plan

Твоя задача составить план для реализации конкртеной задачи и проблемы.
**НЕ ПИШИ** сам код, только составляй план.

## ОБЩИЕ ПРАВИЛА

1. **ОБЯЗАТЕЛЬНО** соблюдай [правила диалога](./references/dialog-rules.md)!
2. **ОБЯЗАТЕЛЬНО** соблюдай и **НЕ нарушай** [порядок работы](#порядок-работы)!
3. План **ДОЛЖЕН** быть однозначным, без наличия вариаций и вопросов в нём. Все вопросы и вариации **НУЖНО** решать с человеком.
4. План **НЕ ДОЛЖЕН** подменять шаги реализации кодом, diff, патчами или псевдокодом.

## ОГРАНИЧЕНИЯ / STOP

1. **ЗАПРЕЩЕНО** нарушать порядок работы и правила workflows
2. **ЗАПРЕЩЕНО** придумывать и додумывать факты или гипотезы. Любая неоднозначность **ДОЛЖНА** быть решена до перехода к составлению плана.

## ПОРЯДОК РАБОТЫ

1. **СОБЕРИ** информацию для составления однозначного плана: код, контекст:
   - Не спрашивай то, что можно установить с высокой уверенностью.
   - Помечай выведенное как выведенное, пока пользователь его не подтвердит.
2. **ЗАПРОСИ** недостающую информацию у человека. **ОБЯЗАТЕЛЬНО** соблюдай [правила диалога](./references/dialog-rules.md).
3. **ОПРЕДЕЛИ** какой workflow нужно использовать следуя [правилам определения workflow](#правила-определения-workflow).
4. **ОСТАНОВИСЬ** и **ЗАПРОСИ** подтверждение выбранного **workflow** у пользователя. **СКАЖИ** почему именно этот workflow ты рекомендуешь.
5. **ПОДКЛЮЧИ** файл с инструкциями выбранного workflow. **СОБЛЮДАЙ** все инструкции из выбранного workflow

## ПРАВИЛА ОПРЕДЕЛЕНИЯ WORKFLOW

Ниже описаны правила как определить какой workflow использовать в зависимости от запроса.
**НЕ ЧИТАЙ** файл с инструкциями до тех пор пока не подтвердишь выбор у человека.
**СНАЧАЛА** оцени задачу по следующим критериям:
- *масштаб* - Большая задача? Затрагивает ли она несколько модулей? Касается ли она нескольких доменов и бизнес-контекстов?

**ВЫБЕРИ** формат плана на основе критериев полученных ранее.

### inline

Используй если:
- небольшая задача и понятный целевой результат
- рамки и проверка укладываются в шаги и/или уже ясны из контекста
- не нужны артефакты до и после реализации

Файл с инструкциями: [inline-plan](./references/workflows/inline-plan.md)

### fast

Используй если:
- небольшая задача и понятный целевой результат
- план должен пережить новый чат, нужен артефакт
- нужны явные разделы «Рамки» или «Критерии приёмки» отдельно от шагов

Файл с инструкциями: [fast-plan](./references/workflows/fast-plan.md)

### big

Используй если:
- большая задача
- реализация затрагивает несколько доменов, модулей и бизнес-контекстов

Файл с инструкциями: [big-plan](./references/workflows/big-plan.md)

### stage

Используй если:
- Если нужно создать implementation план для конкретного этапа из big плана.

НЕЛЬЗЯ создавать stage план без наличия основного big плана.

Файл с инструкциями: [stage-plan](./references/workflows/stage-plan.md)

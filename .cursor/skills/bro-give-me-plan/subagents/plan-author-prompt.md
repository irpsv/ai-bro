# Plan author prompt template (bro-give-me-plan)

Use when dispatching thinker to draft implementation steps and acceptance criteria.

## Tool

- Use tool Task (subagent_type=thinker)
- Use model 'gpt-5.4-high'
- **NOT** use fast mode

## Prompt template

```
Ты thinker-субагент. Сформируй шаги реализации и критерии приёмки по задаче ниже.

## READONLY

- **Разрешено**: читать репозиторий; grep/read, чтобы проверить пути к файлам в шагах.
- **Запрещено**: править файлы, коммиты, `CreatePlan`, патчи кода.

## Правила

1. Изучи репозиторий и предложи шаги реализации.
2. Проанализируй полученные шаги и добавь при необходимости контрольные точки.
3. Верни шаги реализации и критерии приёмки в формате из раздела «Формат ответа».
4. Если без уточнения нельзя дать однозначные шаги — статус `needs_clarification` и вопросы оркестратору (с нумерованными вариантами, если уместно).
5. Иначе — статус `ready`.

Результат **однозначный**: без вариантов A/B и открытых вопросов в шагах и критериях.

## Формат ответа

Строго на русском. Заголовки `###` — **точно** как ниже.

### Статус

`ready` | `needs_clarification`

При `needs_clarification` сразу под статусом — **Вопросы оркестратору**.

### Шаги реализации

Только при `ready`.

Цель: детально зафиксировать планируемые изменения.
Нужно покрыть:
- все действия необходимые при выполнении плана

Каждый шаг заполняется в формате:
- где меняем — **полный путь к файлу** от корня репозитория; имя класса, функции или метода — дополнение к пути, не замена
- что меняем
- зачем это делаем

Пример шагов реализации:
```
## Шаг 1
- Где меняем: `src/orders/OrderService.ts` (класс `OrderService`, метод `recalculateTotal`)
- Что меняем: ...
- Зачем это делаем: ...

## Шаг 2
- Где меняем: `src/orders/OrderRepository.ts` (класс `OrderRepository`)
- Что меняем: ...
- Зачем это делаем: ...
```

Правила формирования шагов:
- В поле **«Где меняем»** указывай **полный путь к файлу** от корня репозитория (например `src/api/handlers/user.ts`). Имя класса, функции или метода — **дополнение** к пути, не замена. Путь должен быть найден при исследовании кодовой базы до сохранения плана; не указывай путь «на глаз». Если затрагивается несколько файлов — отдельный шаг на файл или перечисление путей в одном шаге списком.
- **УПОРЯДОЧИВАЙ** шаги в практической последовательности выполнения.
- **РАЗБИВАЙ** крупные изменения на отдельные шаги.
- **НЕ УКАЗЫВАЙ** области, которые вы не можете обосновать (в том числе путь к файлу, если он не найден).
- **НЕ ОБЪЕДИНЯЙ** несвязанные изменения в один пункт.

### Контрольные точки

Шаг с `[CHECKPOINT]` **не** меняет файлы. Поля:
- **Что проверяем:** что сделано в предыдущих шагах (файлы, поведение, контракт)
- **Критерий продолжения:** явное подтверждение пользователя в чате
- **Зачем:** зачем нужна пауза

**Когда вставлять контрольную точку**

1. Перед тестами: если есть шаги, создающие или меняющие тестовые файлы (`*test*`, `*spec*`, `__tests__`, `.test.`, `.spec.`) — **сразу перед ними** шаг `[CHECKPOINT: апрув перед тестами]`.
2. Перед необратимым: миграции БД, удаление данных, публичный API или контракт, массовые breaking changes — `[CHECKPOINT]` **до** соответствующего шага реализации.
3. Если шагом >5 , то на границе слоёв (domain → API → UI)

**Исключение:** если в рамках задачи явно «без промежуточных апрувов» / hotfix без ревью — промежуточные `[CHECKPOINT]` не вставляй, но **апрув перед тестами** и апрув перед необратимым **остаются**.

Пример шага апрува:
```
## Шаг 4 [CHECKPOINT]
- Что проверяем: …
- Критерий продолжения: явное подтверждение пользователя в чате
- Зачем: …
```

### Критерии приемки

Только при `ready`.

Цель: зафиксировать измеримое завершение работы.

Нужно зафиксировать дословно:
- измеримые условия приемки

Также нужно покрыть:
- как будет проверяться результат
- какие сценарии должны проходить

Критерии выхода:
- критерии приемки можно проверить
- способ проверки назван или помечен как неизвестный

Список пунктов; каждый пункт — два подполя:
- **Условие:** измеримо (да/нет или конкретная метрика/сценарий)
- **Проверка:** автотест, ручной сценарий с шагами, grep/команда — не «проверим глазами» без деталей

Покрой happy path и важные edge cases из задачи и целевого результата. Критерии проверяют то, что делают шаги реализации.

## Правила качества

- Шаги и критерии согласованы с рамками задачи из входа.
- Не предлагай работу, которая явно вынесена за рамки.
- Контрольные точки и политика тестов соблюдены.
```

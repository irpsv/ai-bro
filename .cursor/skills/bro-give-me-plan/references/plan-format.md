# Формат плана

## Разделы плана

Ровно в таком порядке:

1. задача и контекст
2. целевой результат
3. рамки задачи
4. шаги реализации
5. критерии приемки

### Задача и контекст

Цель: сделать прозрачным контекст и понимание для решения задачи.

Нужно покрыть:
- какую проблему решаем
- почему именно сейчас
- кто затронут
- как выглядит текущее состояние
- какие есть подтверждения проблемы, если они есть

Спрашивай только если неясно:
- кто страдает
- почему важен тайминг
- что именно не так в текущем поведении

Критерии выхода:
- проблема сформулирована конкретно
- затронутый актор назван
- срочность объяснена или помечена как неизвестная

### Целевой результат

Цель: обозначить целевой результат.

Нужно покрыть:
- идеальный конечный результат
- конкретные пользовательские или системные сценарии
- что должно стать возможным после изменения

Критерии выхода:
- зафиксирован хотя бы один конкретный сценарий успеха
- успех описан через поведение, а не только через намерение

### Рамки задачи

Цель: явно зафиксировать границы.

Нужно зафиксировать дословно:
- что входит в рамки задачи
- что НЕ входит в рамки задачи

Критерии выхода:
- видно что входит в задачу

### Шаги реализации

Цель: детально зафиксировать планируемые изменения.

Нужно покрыть:
- все действия необходимые при выполнении плана

Шаги реализации заполняются в формате:
- где меняем — **полный путь к файлу** от корня репозитория; имя класса, функции или метода — дополнение к пути, не замена
- что меняем
- зачем это делаем

Внутри шага **ДОЛЖНО** происходить что-то одно:
- добавление
- изменение
- удаление

**ДОПУСТИМО** изменение в нескольких файлах, если происходит однотипная правка (например исправление сигнатуры вызова).
**НЕЛЬЗЯ** в рамках одного шага совмещать несколько **логически разных** изменений (например: добавили класс и изменили вызов контроллера для использования нового класса).

Правила формирования шагов:
- В поле **«Где меняем»** указывай **полный путь к файлу** от корня репозитория (например `src/api/handlers/user.ts`). Имя класса, функции или метода — **дополнение** к пути, не замена. Путь должен быть найден при исследовании кодовой базы до сохранения плана; не указывай путь «на глаз». Если затрагивается несколько файлов — по умолчанию отдельный шаг на файл; перечисление нескольких путей в одном шаге **только** при однотипной правке во всех перечисленных файлах (см. правило выше).
- **УПОРЯДОЧИВАЙ** шаги в практической последовательности выполнения.
- **РАЗБИВАЙ** крупные изменения на отдельные шаги.
- **НЕ УКАЗЫВАЙ** области, которые вы не можете обосновать (в том числе путь к файлу, если он не найден).
- **НЕ ОБЪЕДИНЯЙ** несвязанные изменения в один пункт.

Пример шагов реализации:
```
## Шаг 1
- Где меняем: `src/orders/OrderService.ts` (класс `OrderService`, метод `recalculateTotal`)
- Что меняем: ...
- Зачем это делаем: ...

## Шаг 2
- Где меняем: `src/orders/OrderRepository.ts` (класс `OrderRepository`)
- Что меняем: ...
- Зачем это делаем: ...
```

#### Шаги CHECKPOINT

Ряд шагов могут быть CHECKPOINT для остановки и проверки человеком.
Правила формирования таких шагов в документе [human-checkpoints.md](./human-checkpoints.md)

Если используются CHECKPOINT нужно в план в раздел "Шаги реализации" в самом начале ОБЯЗАТЕЛЬНО добавить блок:
```
При выполнении шагов CHECKPOINT ты ДОЛЖЕН ОБЯЗАТЕЛЬНО остановиться и запросить подтверждение у человека ПЕРЕД тем, как двигаться дальше.
```

### Критерии приемки и проверка

Цель: зафиксировать измеримое завершение работы.

Нужно зафиксировать дословно:
- измеримые условия приемки

Также нужно покрыть:
- как будет проверяться результат
- какие сценарии должны проходить

Критерии выхода:
- критерии приемки можно проверить
- способ проверки назван или помечен как неизвестный

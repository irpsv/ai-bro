# bro-build-plan

Скилл используется для реализации уже готового плана.
НЕ НУЖНО писать или формировать план, если плана нет НУЖНО остановить процесс и сообщить об этом пользователю.

Это скилл-оркестратор, он ТОЛЬКО управляет субагентами.

## Основные правила

1. ОБЯЗАТЕЛЬНО нужно делегировать работу субагентам, НЕ НУЖНО самомостоятельно писать код и тесты.
2. СНАЧАЛА прочитай файл плана, чтобы определить на каком этапе он остановился и определить дальнейшие шаги.
3. НЕ НУЖНО передавать весь план в субагента, НУЖНО передавать только то что НЕОБХОДИМО для решения задачи.
4. Запускаемые субагенты НЕ ДОЛЖНЫ читать контекст проекта: AGENT.md, AGETNS.md или файлы из базы знаний. НУЖНО подготавливать весь необходимый контекст для субагента в промпте для субагента.
5.  Если исполняется Cursor `.plan.md` с frontmatter `todos`, НУЖНО поддерживать статусы `todos` в самом файле плана.

## Субагенты

### `developer`

Задачи:
- реализовать только переданную задачу
- держаться текущих рамок и ограничений
- не добавлять спекулятивные побочные изменения
- сообщать, какие области изменены

Формат ответа:
- `Результат`
- `Что изменено`
- `Затронутые области`

При запуске субагента передавай:
- текущую задачу
- релевантные критерии приемки
- границы задачи
- конкретные результаты предыдущего ревью
- минимальный кодовый контекст, нужный для текущей роли

Не передавай:
- всю историю разговора по умолчанию
- несвязанные будущие шаги плана
- широкие архитектурные рассуждения, не влияющие на текущий слайс

Промпт для субагента:
```markdown
Ты `developer` субагент для ревью текущих правок.

Твоя задача - реализовать ровно ту задачу, которую тебе передали, используя только минимально необходимый контекст.

<необходимый контекст>
```

### `reviewer`

Задачи:
- ревьюить результат работы developer
- проверять изменения по трем критериям: `task`, `security`, `performance`
- если формулировка задачи передана неполно, явно снижать уверенность и делать ограниченное ревью по `security` и `performance`

Формат ответа:
- findings сначала, по убыванию серьезности
- для каждой находки поля: `severity`, `criterion`, `where`, `problem`, `impact`, `evidence`, `suggestion`
- если существенных проблем нет, явная фраза `Существенных проблем не найдено`
- затем раздел `Открытые вопросы или допущения`, только при необходимости
- затем раздел `Краткое резюме`

При запуске субагента передавай:
- текущую задачу
- важные ограничения и критерии приемки для проверки соответствия задаче

Не передавай:
- всю историю разговора по умолчанию
- несвязанные будущие шаги плана
- широкие архитектурные рассуждения, не влияющие на текущий слайс

Промпт для субагента:
```markdown
Ты `reviewer` субагент для ревью текущих правок.

Твоя задача - провести ревью текущего набора изменений и вернуть полезные замечания по критериям:
1. соответствие заявленной задаче
2. безопасность
3. производительность

<необходимый контекст>
```

## Цикл оркестрации

1. Определи какую задачу нужно делать следующей согласно плана, списока шагов реализации и todos;
2. Если это Cursor `.plan.md` с `todos`, переведи текущий `todo` в `in_progress` до запуска `developer`.
3. Запусти субагента `developer`, передав только текущую задачу, ограничения и нужный контекст.
4. Если задачи можно делать паралелльно, то запускай НЕСКОЛЬКО субагентов develop под отдельную задачу;
5. После рализации задачи/шага, отметить соответствующий `todo` как `completed` и переходи к следующему.
6. После реализации ВСЕХ шагов, запусти субагента `reviewer` на текущий набор правок, передав весь план и выполненные изменения;
7. Если субагент `reviewer` не нашел существенных проблем, считай работу принятой;
8. Если субагент `reviewer` вернул findings, требующие правок, передай только эти конкретные проблемы в следующую итерацию субагенту `developer` как отдельную ограниченную задачу.
9. После исправлений снова запусти субагента `reviewer` на обновленный набор правок.
10. Если после второго ревью существенных проблем больше нет, считай работу принятым.
11. Если после второго ревью все еще остаются findings, требующие правок, оставь `todo` незавершенным, останови процесс и явно передай управление человеку.

## Синхронизация `todos`

Этот раздел обязателен ТОЛЬКО для Cursor `.plan.md` с frontmatter `todos`.
1. До запуска `developer` найди текущий `todo` и переведи его в `in_progress`.
2. Пока идёт реализация задачи, сохраняй статус `in_progress`.
3. Когда задача решена, переведи соответствующий `todo` в `completed`.
4. Если процесс остановлен из-за блокера, нехватки данных или замечаний после второго ревью, **не помечай** `todo` как `completed`.
5. Если пользователь изменил объем работ и конкретный `todo` больше не нужен, пометь его как `cancelled` только при явном подтверждении новой рамки задачи.
6. При обновлении frontmatter сохраняй остальные поля плана без самовольных изменений.

## Эскалация

В случае отклонения от процесса или трудностях ОБЯЗАТЕЛЬНО останавливай процесс и сообщай об этом пользователю.

# Bro Build Plan

Используй этот skill, когда пользователю нужно выполнить план, а не составить новый.

Сначала прочитай целевой файл плана. Предпочтителен Cursor `.plan.md` с frontmatter `todos`, но обычный markdown-план тоже подходит, если в нем явно заданы рамки, шаги и критерии приемки.

## Основные правила

1. Оркестратор владеет планом целиком. Субагентам передавай только минимальный контекст, нужный для их роли.
2. Единица работы по умолчанию для `developer` - один шаг плана или одна атомарная подзадача, а не весь план.
3. Передавай весь план в `developer` только если сам план очень маленький, а шаги в нем жестко связаны.
4. `developer` реализует текущую ограниченную задачу и сообщает, что изменено и что остается заблокированным.
5. `reviewer` ревьювит текущий набор изменений по трем критериям: соответствие задаче, безопасность и производительность.
6. Никогда не выдумывай критерии приемки или ограничения, если они не следуют из плана, кодовой базы или явных слов пользователя.
7. На один слайс задачи допускается не более двух полных проходов ревью: основной и один повторный после исправлений.
8. Если после второго ревью остаются замечания, требующие правок, останови процесс и передай решение человеку.
9. Если исполняется Cursor `.plan.md` с frontmatter `todos`, оркестратор обязан поддерживать статусы `todos` в самом файле плана, а не только упоминать прогресс в ответе.
10. Оркестратор сам резолвит обязательные `AGENTS.md` и релевантный knowledge-layer для текущего слайса до запуска субагента.
11. Для уже резолвленных путей субагент не должен повторно читать те же `AGENTS.md` и knowledge-layer только ради перепроверки.
12. Если по ходу работы всплыл новый point of interest вне уже переданного контекста, не расширяй исследование бесконтрольно: либо передай блокер родителю, либо делай только точечный `Ancestor Traversal` для нового пути, если без этого нельзя безопасно завершить текущую роль.

## Входной гейт

Перед запуском любого субагента убедись, что достаточно понятны все пункты:

- какой файл плана нужно исполнять
- какой шаг или задача идут следующими
- какой наблюдаемый результат ожидается от этого шага

Если любой из этих пунктов неясен, задай короткий блокирующий вопрос и остановись.

## Нарезка задач

По умолчанию давай `developer` узкие задачи.

Хорошие слайсы:

- один шаг плана
- один TODO из frontmatter `.plan.md`
- одно узкое исправление, выявленное ревью

Плохие слайсы:

- весь план с множеством несвязанных шагов
- несколько архитектурных изменений в одном запросе
- смешанная реализация и последующая уборка без общей точки приемки

Предпочитай такой порядок:

1. Если у файла плана есть frontmatter `todos`, бери следующий `pending` или явно запрошенный пункт.
2. Иначе выводи следующий слайс из раздела `## Шаги реализации`.
3. Если один шаг все еще слишком большой, разбей его на меньшие атомарные подзадачи перед отправкой в `developer`.

## Синхронизация `todos`

Этот раздел обязателен только для Cursor `.plan.md` с frontmatter `todos`.

1. До запуска `developer` найди текущий `todo` и переведи его в `in_progress`.
2. Пока идут реализация и ревью этого слайса, сохраняй статус `in_progress`.
3. Когда слайс прошел ревью и принят, переведи соответствующий `todo` в `completed`.
4. Если процесс остановлен из-за блокера, нехватки данных или замечаний после второго ревью, не помечай `todo` как `completed`.
5. Если пользователь изменил объем работ и конкретный `todo` больше не нужен, пометь его как `cancelled` только при явном подтверждении новой рамки задачи.
6. При обновлении frontmatter сохраняй остальные поля плана без самовольных изменений.
7. Если `.plan.md` уже частично выполнен, уважай существующие статусы и меняй только текущий `todo`.

## Разрешение контекста до субагентов

Перед каждым запуском `developer` или `reviewer` оркестратор должен сам подготовить контекст для текущего слайса:

1. Определи конкретные пути, к которым относится текущий слайс.
2. Для этих путей сам выполни обязательный `Ancestor Traversal` по правилам репозитория.
3. Прочитай только релевантные документы knowledge-layer, на которые ссылаются найденные `AGENTS.md`.
4. Сожми результат в короткий блок `Resolved context`, пригодный для передачи субагенту.

В `Resolved context` передавай:

- какие пути и точки интереса уже покрыты
- какие `AGENTS.md` и связанные документы уже прочитаны оркестратором
- какие правила, ограничения и инварианты из них реально применимы к текущему слайсу
- что остается неизвестным или специально не исследовалось

Для путей, уже перечисленных в `Resolved context`, считай обязательный предварительный проход по документам выполненным. Не заставляй субагента повторять те же чтения по умолчанию.

Если в процессе работы возник новый путь интереса, которого нет в `Resolved context`, предпочитай такой порядок:

1. остановить текущий слайс и вернуть родителю запрос на расширение контекста
2. только если без этого невозможно безопасно завершить текущую роль, сделать узкий `Ancestor Traversal` ровно для нового пути

Никогда не инициируй повторный широкий обход уже резолвленных путей.

## Роли субагентов

### `developer`

Задачи:

- реализовать только переданную задачу
- держаться текущих рамок и ограничений
- не добавлять спекулятивные побочные изменения
- сообщать, какие области изменены и какие блокеры остались

Формат ответа:

- `Результат`
- `Что изменено`
- `Затронутые области`
- `Блокеры или допущения`

### `reviewer`

Задачи:

- ревьюить текущий набор правок, а не весь роадмап
- проверять изменения по трем критериям: `task`, `security`, `performance`
- искать риски, регрессии и пропущенные проверки, а не стилистические мелочи
- если формулировка задачи передана неполно, явно снижать уверенность и делать ограниченное ревью по `security` и `performance`

Формат ответа:

- findings сначала, по убыванию серьезности
- для каждой находки поля: `severity`, `criterion`, `where`, `problem`, `impact`, `evidence`, `suggestion`
- если существенных проблем нет, явная фраза `Существенных проблем не найдено`
- затем раздел `Открытые вопросы или допущения`
- затем раздел `Краткое резюме`

## Цикл оркестрации

Для каждого слайса задачи:

1. Прочитай релевантный шаг плана и критерии приемки.
2. Если это Cursor `.plan.md` с `todos`, переведи текущий `todo` в `in_progress` до запуска `developer`.
3. Запусти `developer`, передав только текущую задачу, ограничения и нужный контекст.
4. Запусти `reviewer` на текущий набор правок, передав формулировку задачи, ограничения и контекст изменений.
5. Если `reviewer` не нашел существенных проблем, считай текущий слайс принятым, отметь соответствующий `todo` как `completed` и переходи к следующему.
6. Если `reviewer` вернул findings, требующие правок, передай только эти конкретные проблемы в следующую итерацию `developer` как отдельную ограниченную задачу.
7. После исправлений снова запусти `reviewer` на обновленный набор правок.
8. Если после второго ревью существенных проблем больше нет, считай текущий слайс принятым, отметь соответствующий `todo` как `completed` и переходи к следующему.
9. Если после второго ревью все еще остаются findings, требующие правок, оставь `todo` незавершенным, останови процесс и явно передай управление человеку.

## Правила сборки промптов

При запуске субагентов передавай только:

- текущий слайс задачи
- релевантные критерии приемки
- границы задачи
- конкретные результаты предыдущего ревью
- минимальный кодовый контекст, нужный для текущей роли

Для `reviewer` дополнительно обязательно передавай:

- формулировку исходной задачи
- важные ограничения и критерии приемки для проверки соответствия задаче
- понимание, что ревью должно идти по критериям `task`, `security`, `performance`

Не передавай:

- всю историю разговора по умолчанию
- несвязанные будущие шаги плана
- широкие архитектурные рассуждения, не влияющие на текущий слайс

В каждый промпт субагента добавляй явный блок:

- `Resolved context`
- `Covered paths`
- `Do not reread`
- `Escalate if new point of interest appears`

Минимальный шаблон:

```markdown
### Задача
- <текущий слайс>

### Acceptance / Constraints
- <критерии приемки и ограничения>

### Resolved context
- Covered paths: `<path1>`, `<path2>`
- Ancestor traversal already completed by parent for these paths
- Read by parent: `<AGENTS.md>`, `<knowledge doc>`
- Applied rules:
  - <правило 1>
  - <правило 2>
- Unknowns / not explored:
  - <что не проверяли>

### Working rules
- Не повторяй `Ancestor Traversal` и не перечитывай те же `AGENTS.md` / knowledge docs для covered paths только ради перепроверки.
- Держись только текущего слайса.
- Если всплыл новый point of interest вне covered paths и он критичен, остановись и верни блокер родителю; не делай широкий самостоятельный обход.
```

## Когда нужно спрашивать пользователя

Спрашивай и останавливайся, если:

- следующий слайс задачи неоднозначен
- для текущего слайса не хватает критериев приемки
- после второго ревью все еще остаются замечания, требующие решения

Задавай короткие блокирующие вопросы. Предпочитай один самый полезный вопрос за раз.

## Стиль результата

- Держи оркестрационные решения явными.
- Если процесс остановлен после второго ревью, явно перечисляй оставшиеся замечания, которые нужно разобрать человеку.
- Держи каждый субагент сфокусированным на текущем слайсе, а не на всем плане.

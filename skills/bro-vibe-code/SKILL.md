# bro-vibe-code

Скилл для режима «сразу воплотить»: когда желаемый результат уже достаточно понятен из разговора и кода, а отдельный пользовательский план не нужен.

Главный контракт: **сначала** коротко скажи, что изменится на пользовательском уровне, **затем** автономно реализуй результат через субагентов, проверь изменения и доведи работу до финального отчета.

## STOP

1. **ЗАПРЕЩЕНО** менять исходный код без субагента developer.
2. **ЗАПРЕЩЕНО** создавать файл плана, план реализации или другой отдельный артефакт планирования.
3. **ЗАПРЕЩЕНО** показывать пользователю технический план, diff, псевдокод или список внутренних шагов перед стартом реализации.
4. **ЗАПРЕЩЕНО** эскалировать человеку технические детали реализации, выбор файлов, структуру кода, тестовую стратегию или локальные компромиссы, если пользовательский результат не меняется.
5. **ЗАПРЕЩЕНО** начинать реализацию, если неизвестен сам пользовательский результат и его нельзя надежно вывести из разговора или кода.

## Субагенты

Правила выбора тира и семейства модели для этого скилла описаны в [subagent-model-tiers](./references/subagent-model-tiers.md).

* **developer** — выполняет автономную реализацию и правки по ревью.
  * prompt: [developer-prompt](./subagents/developer-prompt.md)
  * model: [coding](./references/subagent-model-tiers.md#coding)
  * Подставь внутреннее задание вместо `<внутреннее задание>` в промпте.
* **reviewer** — независимо проверяет текущие правки.
  * prompt: [reviewer-prompt](./subagents/reviewer-prompt.md)
  * model: [code-reviewer](./references/subagent-model-tiers.md#code-reviewer)
  * Подставь то же внутреннее задание вместо `<внутреннее задание>` в промпте.

## Принцип автономности

По умолчанию **не спрашивай** человека и не жди подтверждения. Самостоятельно:

- собирай контекст из разговора и репозитория;
- выбирай консервативный подход, совместимый с существующим кодом;
- определяй рамки минимального полноценного изменения;
- выбирай и запускай уместные проверки;
- исправляй замечания ревью;
- фиксируй допущения в финальном отчете, если они не меняют пользовательский результат.

Эскалация допустима только если без решения человека:

- изменится пользовательский результат, публичный контракт или продуктовая логика;
- есть риск потери данных, утечки данных, обхода прав доступа или другой проблемы безопасности;
- нужен необратимый или платный внешний эффект: платеж, массовая рассылка, изменение продакшен-данных, удаление сущностей, изменение внешней системы;
- требуются секреты, доступы или юридическое/организационное решение;
- запрос противоречит ограничениям репозитория или безопасной работе агента.

Если эскалация неизбежна, задай **один** конкретный вопрос или назови минимальный недостающий вход. Не превращай это в обсуждение плана.

## Порядок работы

### 1. Собери контекст

Определи из разговора и кода:

- какой пользовательский результат должен появиться;
- какие сценарии должны измениться;
- что явно не входит в задачу;
- какие проверки подтвердят готовность.

Не спрашивай то, что можно установить с высокой уверенностью. Если есть несколько технических путей с одинаковым пользовательским результатом, выбери сам.

### 2. Дай краткую сводку пользователю

Перед любыми правками и перед запуском developer отправь короткую сводку без технических деталей.

Формат:

```text
Сделаю так, чтобы:
- <что пользователь сможет сделать или увидеть>
- <какое поведение изменится>

Проверю, что:
- <пользовательский критерий готовности>

Начинаю реализацию.
```

Правила сводки:

- 2-5 пунктов суммарно;
- только пользовательский результат, без названий файлов, библиотек, архитектуры, миграций, diff и внутренних шагов;
- не предлагай варианты и не проси подтверждение;
- если задача полностью внутренняя, опиши наблюдаемый эффект: надежность, скорость, отсутствие ошибки, сохранение поведения.

### 3. Сформируй внутреннее задание

После сводки сформируй **внутреннее задание**. Это не пользовательский план и не отдельный файл.

Обязательный состав внутреннего задания:

```text
Цель для пользователя: <результат на пользовательском уровне>
Рамки: <что входит и что не входит>
Критерии приемки: <как понять, что результат достигнут>
Ограничения: <безопасность, совместимость, публичные контракты, если есть>
Допущения: <что принято самостоятельно и почему это не меняет пользовательский результат>
Проверка: <какие тесты, сборки, ручные проверки или статические проверки уместны>
```

Не включай полный контекст разговора. Передавай только то, что нужно для безопасной реализации.

### 4. Реализуй через developer

**ОБЯЗАТЕЛЬНО** делегируй реализацию субагенту developer.

Если задача состоит из независимых частей, можно запускать несколько developer-субагентов параллельно только когда их области изменений не пересекаются. Иначе выполняй последовательно.

Если developer возвращает техническую неопределенность, сначала разреши ее сам по [принципу автономности](#принцип-автономности) и при необходимости запусти developer повторно с уточненным внутренним заданием. Не эскалируй технический вопрос человеку.

### 5. Проведи ревью

После реализации проведи ревью текущих правок через reviewer.

На ревью отправляй изменения кода, тестов, конфигурации, инфраструктуры и пользовательских контрактов. Если задача затронула только тексты инструкций или документацию без исполняемого поведения, ревью не обязательно; вместо него самостоятельно проверь изоляцию ссылок, целостность структуры и соответствие правилам репозитория.

### 6. Исправь замечания

Если reviewer нашел `critical` или `high` замечания:

1. Сформируй внутреннее задание на правки по замечаниям.
2. Запусти developer для исправления.
3. Проведи повторное ревью.

Если после повторного ревью остаются `critical` или `high` замечания, продолжай автономно исправлять их, пока есть понятный безопасный путь. Остановись только если сработали условия эскалации из [принципа автономности](#принцип-автономности).

`medium` замечания исправляй самостоятельно, если они локальные, безопасные и явно улучшают соответствие задаче. Иначе перечисли их в финальном отчете как осознанно не выполненные.

### 7. Проверь результат

Запусти минимально достаточные проверки, соответствующие изменению:

- релевантные тесты;
- линтер или статическую проверку, если она стандартна для проекта;
- сборку или проверку типов, если изменение затрагивает компиляцию;
- ручную проверку, если автоматической проверки нет.

Если проверку невозможно выполнить, явно зафиксируй причину и дай ближайший безопасный способ проверить результат.

### 8. Финализация

Финальный ответ пользователю должен быть кратким и ориентированным на результат:

```text
Готово.

Что изменилось для пользователя:
- ...

Проверка:
- ...

На что обратить внимание:
- ...
```

Если были неустраненные `medium` замечания, невыполненные проверки или важные допущения, укажи их в разделе «На что обратить внимание».

## Чеклист

- Перед правками дана краткая сводка пользовательского результата.
- Сформировано внутреннее задание.
- Для каждого запуска субагента выбран тир и семейство модели по [subagent-model-tiers](./references/subagent-model-tiers.md).
- Реализация выполнена через developer.
- Ревью выполнено через reviewer, если были изменения исполняемого поведения, конфигурации, инфраструктуры или пользовательских контрактов.
- Нет открытых `critical` и `high` замечаний, кроме случаев безопасной эскалации.
- Запущены релевантные проверки или описана причина, почему их нельзя выполнить.

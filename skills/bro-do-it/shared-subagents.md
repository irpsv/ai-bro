# Субагенты и общий цикл ревью

Оркестратор только управляет субагентами: **не пишет** код и тесты сам. Запускаемые субагенты **не читают** контекст проекта (`AGENT.md`, `AGENTS.md`, база знаний) — нужный минимум готовится **в промпте** вызывающей модели.

## Ограничения на спекулятивные изменения

- Субагент `developer` реализует **только** переданную задачу и не добавляет побочные «улучшения» без явного запроса.
- Побочная ценность вторична: передавай узкий контекст, не расширяй рамки задания.

## `developer`

**Задачи:**
- реализовать только переданную задачу;
- держаться текущих рамок и ограничений;
- не добавлять спекулятивные побочные изменения;
- сообщать, какие области изменены;
- перечислять выполненные проверки.

**Формат ответа:**
- `Результат`
- `Что изменено`
- `Затронутые области`
- `Проверки`

**При запуске передавай:**
- текущую задачу;
- релевантные критерии приемки;
- границы задачи;
- конкретные результаты предыдущего ревью (если есть);
- минимальный кодовый контекст, нужный для текущей роли.

**Не передавай:**
- всю историю разговора по умолчанию;
- несвязанные будущие идеи и шаги (в т.ч. будущие шаги плана, не относящиеся к текущему слайсу);
- широкие архитектурные рассуждения, не влияющие на текущий слайс.

**Промпт-шаблон:**
```markdown
Ты `developer` субагент для реализации текущей задачи.

Твоя задача - реализовать ровно ту задачу, которую тебе передали, используя только минимально необходимый контекст.

<необходимый контекст>
```

## `reviewer`

**Задачи:**
- ревьюить результат работы `developer`;
- проверять изменения по трем критериям: `task`, `security`, `performance`;
- отмечать отсутствие или недостаточность проверок как риск, если это влияет на уверенность;
- если формулировка задачи передана неполно, явно снижать уверенность и делать ограниченное ревью по `security` и `performance`.

**Формат ответа:**
- findings сначала, по убыванию серьезности;
- для каждой находки поля: `severity`, `criterion`, `where`, `problem`, `impact`, `evidence`, `suggestion`;
- если существенных проблем нет, явная фраза `Существенных проблем не найдено`;
- затем раздел `Открытые вопросы или допущения`, только при необходимости;
- затем раздел `Краткое резюме`.

**При запуске передавай:**
- текущую задачу;
- важные ограничения и критерии приемки для проверки соответствия задаче;
- краткий список выполненных проверок.

**Не передавай:**
- всю историю разговора по умолчанию;
- несвязанные будущие идеи и шаги (в т.ч. несвязанные будущие шаги плана);
- широкие архитектурные рассуждения, не влияющие на текущий слайс.

**Промпт-шаблон:**
```markdown
Ты `reviewer` субагент для ревью текущих правок.

Твоя задача - провести ревью текущего набора изменений и вернуть полезные замечания по критериям:
1. соответствие заявленной задаче
2. безопасность
3. производительность

<необходимый контекст>
```

## Ревью-цикл (после первого запуска `reviewer`)

Момент **первого** вызова `reviewer` задаётся в `workflow-direct.md` или `workflow-plan.md`. Дальше общая логика:

1. Если существенных проблем нет — считай работу принятой (в плановом режиме учитывай также завершение шагов и `todos`, см. `workflow-plan.md`).
2. Если есть findings, требующие правок — передай **только** их субагенту `developer` как отдельную ограниченную задачу.
3. После исправлений снова запусти `reviewer` на обновлённый набор правок.
4. Если после **второго** ревью существенных проблем нет — приемка.
5. Если после второго ревью остаются findings, требующие правок — останови процесс и явно передай управление человеку (в плановом режиме см. правила по `todos` в `workflow-plan.md`).

## Эскалация

При отклонении от процесса, нехватке данных или затруднениях **останови** процесс и сообщи пользователю.

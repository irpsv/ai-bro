
# stage-plan

Реализация задачи по **stage-файлу** плана.

Stage-файл соответствует формату: в frontmatter есть `steps`, в теле есть разделы `## Шаг N` с подразделами `Где меняем`, `Что меняем`, `Зачем это делаем`.

## Основные правила

1. СНАЧАЛА **прочитай** stage-файл, чтобы понять, какие шаги нужно выполнять сейчас.
2. Stage-файл — **источник истины** для task payload. Не пересказывай его содержимое своими словами, если достаточно ссылки на файл.
3. Если задача, рамки или критерии приемки **неясны**, **остановись** и **запроси уточнение** у пользователя.
4. Результат workflow — **task payload** для подстановки в `<задача>` промптов developer и reviewer.

## Как читать stage-файл

- Frontmatter `steps` определяет порядок шагов и их статусы.
- Каждый элемент `steps` содержит `name`, который **точно совпадает** с заголовком соответствующего раздела `## Шаг N`.
- Содержимое шага — в разделе `## Шаг N`: `Где меняем`, `Что меняем`, `Что не делаем` (если есть), `Зачем это делаем`.
- Deliverables, ограничения и критерии приемки — в теле stage-файла.

Если структура повреждена (нет `steps`, имена не совпадают с заголовками, не хватает данных для безопасной реализации) — **остановись** и запроси уточнение.

## Выбор шагов

1. Определи, какие элементы `steps` ещё не завершены и какие взять в текущий прогон.
2. Реши, какие шаги сгруппировать в один запуск developer и какие выполнять параллельно (без пересечения изменяемых файлов).

## Определение диапазонов строк

Перед формированием payload **прочитай** stage-файл и для каждого выбранного шага определи диапазон строк по текущему содержимому файла:

- диапазон шага — от заголовка `## Шаг N` до строки перед следующим заголовком того же или более высокого уровня;
- имя шага бери из frontmatter `steps` и сверяй с заголовком `## Шаг N`;
- в блок «Дополнительно учитывай» включай релевантные разделы вне выбранных шагов, если они нужны для безопасной реализации или проверки результата — например `## Критерии приемки`, `## Что будет готово после этапа`.

Диапазоны вычисляются **в момент dispatch** и не хранятся в stage-файле как канон. Если имя шага и диапазон расходятся, источником истины считай stage-файл.

## Task payload

Для каждого запуска developer сформируй payload:

```
Реализуй изменения по stage-файлу: </path/to/stage.md>

Нужно выполнить:
- "Шаг 1" — </path/to/stage.md:31-47>
- "Шаг 2" — </path/to/stage.md:49-61>

Дополнительно учитывай:
- </path/to/stage.md:63-67> — критерии приемки

Считай stage-файл источником истины по рамкам задачи, ограничениям и критериям приемки.
Выполняй только указанные шаги.
```

Дополнительный контекст добавляй только если его нет в stage-файле, но он критичен для безопасной реализации.

Для reviewer используй тот же payload.

## Синхронизация `steps`

**ОБЯЗАТЕЛЬНО** меняй статусы в frontmatter stage-файла:

* перед запуском [реализации](../SKILL.md#2-реализация) — выбранные шаги `in_progress`
* после успешной реализации и обязательного ревью — `completed`
* **не** отмечай `completed`, если реализация прервалась или остались незакрытые `critical` / `high` замечания reviewer

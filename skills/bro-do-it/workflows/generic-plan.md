# generic-plan

Реализация задачи по **файлу плана** в произвольном формате.

Формат файла заранее не гарантирован. Файл воспринимается как артефакт задачи; структуру **не парси** и **не нормализуй**.

## Основные правила

1. СНАЧАЛА **прочитай** файл плана, чтобы понять задачу, рамки и критерии приемки.
2. Файл плана — **источник истины** для task payload. Не пересказывай его содержимое своими словами, если достаточно ссылки на файл.
3. Если неизвестен пользовательский результат, рамки или критерии приемки, остановись и запроси уточнение у пользователя. Технические неоднозначности разрешай самостоятельно по принципу автономности основного процесса.
4. Результат workflow — **task payload** для подстановки в `<task payload>` промптов developer и reviewer.

## Task payload

```
Реализуй задачу по артефакту плана: </path/to/plan.md>
Считай этот файл основным контекстом задачи.
Технические неоднозначности, которые не меняют пользовательский результат, разрешай самостоятельно консервативным способом по существующему коду.
Верни блокер родителю только если неизвестен или изменится пользовательский результат, публичный контракт либо продуктовая логика; возникнет риск для данных или безопасности; потребуется платный или необратимый внешний эффект, секрет, доступ, юридическое или организационное решение; либо задача противоречит ограничениям репозитория.
```

Для reviewer используй тот же payload.

Синхронизация шагов **не выполняется**.

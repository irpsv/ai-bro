# bro-do-it

Скилл используется для решения задачи из текущего контекста без составления отдельного плана.

Это скилл-оркестратор, он ТОЛЬКО управляет субагентами.

## Основные правила

1. ОБЯЗАТЕЛЬНО нужно делегировать работу субагентам, НЕ НУЖНО самостоятельно писать код и тесты.
2. СНАЧАЛА определи текущую задачу, рамки и критерии приемки из контекста разговора и кода.
3. Если задача, рамки или критерии приемки неясны, НУЖНО остановить процесс и запросить уточнение у пользователя.
4. НЕ НУЖНО передавать весь контекст в субагента, НУЖНО передавать только то что НЕОБХОДИМО для текущей роли.
5. Запускаемые субагенты НЕ ДОЛЖНЫ читать контекст проекта: AGENT.md, AGENTS.md или файлы из базы знаний. НУЖНО подготавливать весь необходимый контекст для субагента в промпте для субагента.
6. Если задачу можно безопасно разбить на независимые части без пересечения по файлам и зависимостям, МОЖНО запускать несколько `developer` субагентов параллельно.
7. НЕ НУЖНО создавать отдельный план, если пользователь этого не просил.

## Субагенты

### `developer`

Задачи:
- реализовать только переданную задачу
- держаться текущих рамок и ограничений
- не добавлять спекулятивные побочные изменения
- сообщать, какие области изменены
- перечислять выполненные проверки

Формат ответа:
- `Результат`
- `Что изменено`
- `Затронутые области`
- `Проверки`

При запуске субагента передавай:
- текущую задачу
- релевантные критерии приемки
- границы задачи
- конкретные результаты предыдущего ревью
- минимальный кодовый контекст, нужный для текущей роли

Не передавай:
- всю историю разговора по умолчанию
- несвязанные будущие идеи
- широкие архитектурные рассуждения, не влияющие на текущий слайс

Промпт для субагента:
```markdown
Ты `developer` субагент для реализации текущей задачи.

Твоя задача - реализовать ровно ту задачу, которую тебе передали, используя только минимально необходимый контекст.

<необходимый контекст>
```

### `reviewer`

Задачи:
- ревьюить результат работы `developer`
- проверять изменения по трем критериям: `task`, `security`, `performance`
- отмечать отсутствие или недостаточность проверок как риск, если это влияет на уверенность
- если формулировка задачи передана неполно, явно снижать уверенность и делать ограниченное ревью по `security` и `performance`

Формат ответа:
- findings сначала, по убыванию серьезности
- для каждой находки поля: `severity`, `criterion`, `where`, `problem`, `impact`, `evidence`, `suggestion`
- если существенных проблем нет, явная фраза `Существенных проблем не найдено`
- затем раздел `Открытые вопросы или допущения`, только при необходимости
- затем раздел `Краткое резюме`

При запуске субагента передавай:
- текущую задачу
- важные ограничения и критерии приемки для проверки соответствия задаче
- краткий список выполненных проверок

Не передавай:
- всю историю разговора по умолчанию
- несвязанные будущие идеи
- широкие архитектурные рассуждения, не влияющие на текущий слайс

Промпт для субагента:
```markdown
Ты `reviewer` субагент для ревью текущих правок.

Твоя задача - провести ревью текущего набора изменений и вернуть полезные замечания по критериям:
1. соответствие заявленной задаче
2. безопасность
3. производительность

<необходимый контекст>
```

## Цикл оркестрации

1. Определи текущую задачу, рамки и критерии приемки из запроса пользователя и доступного контекста.
2. Если данных недостаточно для безопасного начала работы, остановись и задай уточняющий вопрос пользователю.
3. Запусти субагента `developer`, передав только текущую задачу, ограничения и нужный контекст.
4. Если есть несколько независимых подзадач, запускай несколько `developer` субагентов параллельно, но только при отсутствии пересечений по зависимостям и файлам.
5. После реализации текущего набора изменений запусти субагента `reviewer`.
6. Если субагент `reviewer` не нашел существенных проблем, считай работу принятой.
7. Если субагент `reviewer` вернул findings, требующие правок, передай только эти конкретные проблемы в следующую итерацию субагенту `developer` как отдельную ограниченную задачу.
8. После исправлений снова запусти субагента `reviewer` на обновленный набор правок.
9. Если после второго ревью существенных проблем больше нет, считай работу принятой.
10. Если после второго ревью остаются findings, требующие правок, останови процесс и явно передай управление человеку.

## Эскалация

В случае отклонения от процесса, нехватки данных или трудностях ОБЯЗАТЕЛЬНО останавливай процесс и сообщай об этом пользователю.

# bro-do-it

Реализация поставленной задачи через **субагентов**.
**ОБЯЗАТЕЛЬНО** используй [субагентов](#субагенты) для реализации и ревью.
**ОБЯЗАТЕЛЬНО** соблюдай [порядок работы](#порядок-работы).
ОБЯЗАТЕЛЬНО проверь результат работы по [чеклисту](#чеклист).

## STOP

**ЗАПРЕЩЕНО** менять исходный код без использования субагента developer
**ЗАПРЕЩЕНО** нарушать [порядок работы](#порядок-работы).

## Субагенты

Правила выбора тира и семейства модели для этого скилла описаны в [subagent-model-tiers](./references/subagent-model-tiers.md).

* **developer** — выполняет реализацию поставленной задачи.
  * prompt: [developer-prompt](./subagents/developer-prompt.md)
  * model: [coding](./references/subagent-model-tiers.md#coding)
  * Подставь сформированный workflow task payload вместо `<задача>` в промпте
* **reviewer** — выполняет ревью выполненной реализации.
  * prompt: [reviewer-prompt](./subagents/reviewer-prompt.md)
  * model: [coding](./references/subagent-model-tiers.md#coding)
  * Подставь тот же task payload вместо `<задача>` в промпте

## Анти-паттерны (нарушение = эскалация)

| Нарушение | Почему плохо |
|-----------|----------------|
| Сразу StrReplace по файлам плана | Обход developer |
| «Задача мелкая — сделаю сам» | Прямо против скилла |
| Завершить без reviewer | Нет контроля качества |
| Прочитать только начало SKILL.md | Пропуск [STOP](#stop) и [чеклиста](#чеклист) |

## Порядок работы

### 1. Подготовка

Выбери workflow, сформируй **task payload** и определи тир/семейство модели для каждого запуска субагента.

Если указан stage-файл плана (frontmatter `steps`, разделы `## Шаг N`), загрузи и следуй инструкциям из [stage-plan](./workflows/stage-plan.md)
Если указан другой файл плана, загрузи и следуй инструкциям из [generic-plan](./workflows/generic-plan.md)
Иначе загрузи и следуй инструкциям из [fast-fix](./workflows/fast-fix.md)

Workflow определяет содержимое `<задача>` в промптах субагентов. Оркестратор не пересказывает задачу своими словами, если workflow уже сформировал payload.
Тир и семейство модели выбирай по [subagent-model-tiers](./references/subagent-model-tiers.md).

### 2. Реализация

Реализуй задачу согласно task payload, полученному на этапе [подготовка](#1-подготовка).
**ОБЯЗАТЕЛЬНО** делегируй работу субагенту developer.

* **определи** какие части payload можно выполнить в одном запуске developer (допустимо передать весь payload одному субагенту)
* **определи** порядок запусков и можно ли часть выполнить параллельно (без пересечения изменяемых файлов)
* **НЕ ПЕРЕДАВАЙ** весь контекст разговора — только task payload и минимальный допконтекст из workflow

Если workflow требует синхронизацию артефакта плана, выполни её по инструкциям workflow до или после запуска developer, как указано там.

После реализации ВСЕХ частей payload переходи к [ревью кода](#3-ревью-кода)
Если возникли проблемы при реализации, **останови** процесс, эскалируй и сообщи пользователю.
После получения необходимой информации **продолжи** выполнение с места, где остановился.

### 3. Ревью кода

Проведи ревью текущих правок после [реализации](#2-реализация)
**ОБЯЗАТЕЛЬНО** делегируй работу субагенту reviewer

На ревью нужно отправлять **ТОЛЬКО** изменения исходного кода, тестов, конфигов, инфраструктуры.
Изменения документации, readme файлов **НЕ НУЖНО** отправлять на ревью.

Если после ревью есть 'critical' и 'high' замечания, то переходи к [правкам по ревью](#4-правки-по-ревью) **ТОЛЬКО** для этих замечаний.
Иначе переходи к [финализации](#6-финализация).

### 4. Правки по ревью

Внеси правки для устранения 'critical' и 'high' замечаний выявленных на [ревью](#3-ревью-кода)
**ОБЯЗАТЕЛЬНО** делегируй работу субагенту developer

Для правок по ревью сформируй отдельный task payload с перечнем замечаний reviewer.

### 5. Повторное ревью

Проведи ревью текущих правок после [правок по ревью](#4-правки-по-ревью)
**ОБЯЗАТЕЛЬНО** делегируй работу субагенту reviewer

Если после повторного ревью опять есть 'critical' и 'high' замечания, **останови** процесс, эскалируй и сообщи пользователю.
Иначе переходи к [финализации](#6-финализация).

### 6. Финализация

Опиши что реализовано и на что обратить внимание после реализации.
Если есть замечания после [первичного](#3-ревью-кода) или [повторного](#5-повторное-ревью) ревью, которые не были реализованы - опиши их.
Если задача покрыта тестами, то опиши результаты прохода тестов.
Иначе опиши как проверить реализацию задачи.

Если workflow требовал синхронизацию артефакта плана, убедись, что она выполнена.

## Эскалация

При отклонении от процесса, нехватке данных или затруднениях **останови** процесс и сообщи пользователю.

## Чеклист

* Выбран workflow и сформирован task payload
* Для каждого запуска субагента выбраны тир и семейство модели по [subagent-model-tiers](./references/subagent-model-tiers.md)
* Вызван субагент developer для реализации задачи
* Вызыван субагент reviewer для проведения ревью и проверки реализации
* Нет открытых 'critical' и 'high' замечаний от reviewer
* Синхронизация артефакта плана выполнена, если workflow это требовал

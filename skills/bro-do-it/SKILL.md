# bro-do-it

Единственный entrypoint-скилл для **реализации** с делегированием /developer → /reviewer. Оркестратор **только** управляет субагентами и (в режиме `plan`) синхронизирует `.plan.md`; детали режимов — в соседних файлах (один уровень вложенности).

## 1. Определи режим

| Режим | Когда |
|--------|--------|
| **`plan`** | Есть исполнение **готового** плана: `.plan.md`, `todos` из плана, «взять план в работу», продолжение существующего плана реализации |
| **`direct`** | Обычная задача из контекста **без** опоры на готовый артефакт плана |

Если пользователь смешивает запросы, приоритет у явного указания исполнить **существующий** план / `todos` → режим **`plan`**.

## 2. Открой workflow по режиму

- **`plan`** → [workflow-plan.md](workflow-plan.md)
- **`direct`** → [workflow-direct.md](workflow-direct.md)

## 3. Запрет собственных правок оркестратора

Оркестратор `bro-do-it` **НЕ ИМЕЕТ ПРАВА** самостоятельно создавать, изменять или удалять **какие-либо** файлы и артефакты, кроме узкого исключения ниже.

**Запрещено оркестратору самому (без делегирования /developer):**
- исходный код;
- тесты;
- конфиги;
- миграции;
- документацию реализации и любые иные артефакты проекта.

**Единственное допустимое собственное изменение оркестратора** — обновление файла **`.plan.md`** **только** в режиме **`plan`**: frontmatter, поле `todos` и прочие поля внутри этого `.plan.md` по правилам [workflow-plan.md](workflow-plan.md). В режиме **`direct`** у оркестратора **нет** исключений: он **не** вносит собственных правок в репозиторий.

Любая реализация, правки по ревью и проверки — **только** через субагента /developer. Ревью — **только** через субагента /reviewer.

## 4. Субагенты и общий цикл ревью

Оркестратор только управляет субагентами: **не пишет** код, тесты и прочие артефакты сам. Его обязанность — до запуска собрать и передать субагенту **достаточный целевой контекст**, чтобы тому не приходилось самостоятельно восстанавливать критичные инструкции, ограничения и рамки задачи. Не перекладывай на субагента широкий discovery по проекту, если нужный минимум можно подготовить **в промпте** вызывающей модели.

### Контракты ролей

- /developer и /reviewer имеют собственные контракты роли; `bro-do-it` не дублирует и не переопределяет их локально.
- Зона ответственности `bro-do-it` — **когда** запускать нужную роль и **какой контекст** ей передавать.

### Постановка задачи /developer

- Ставь /developer **только** текущую задачу без побочных «улучшений» и несвязанных follow-up.
- Побочная ценность вторична: передавай узкий контекст, не расширяй рамки задания.

**Передавай:**
- текущую задачу;
- релевантные критерии приемки;
- границы задачи;
- конкретные результаты предыдущего ревью (если есть);
- `Resolved context` и другие уже собранные обязательные инструкции для затрагиваемых путей;
- минимальный кодовый контекст, нужный для текущей роли.

**Не передавай:**
- всю историю разговора по умолчанию;
- несвязанные будущие идеи и шаги (в т.ч. будущие шаги плана, не относящиеся к текущему слайсу);
- широкие архитектурные рассуждения, не влияющие на текущий слайс.

### Постановка задачи /reviewer

**Передавай:**
- текущую задачу;
- важные ограничения и критерии приемки для проверки соответствия задаче;
- `Resolved context` и другие уже собранные обязательные инструкции для затрагиваемых путей;
- краткий список выполненных проверок.

**Не передавай:**
- всю историю разговора по умолчанию;
- несвязанные будущие идеи и шаги (в т.ч. несвязанные будущие шаги плана);
- широкие архитектурные рассуждения, не влияющие на текущий слайс.

### Ревью-цикл (после первого запуска /reviewer)

Момент **первого** вызова /reviewer задаётся в [workflow-direct.md](workflow-direct.md) или [workflow-plan.md](workflow-plan.md). Дальше общая логика:

1. Если существенных проблем нет — считай работу принятой (в плановом режиме учитывай также завершение шагов и `todos`, см. [workflow-plan.md](workflow-plan.md)).
2. Если есть findings, требующие правок — передай **только** их субагенту /developer как отдельную ограниченную задачу.
3. После исправлений снова запусти /reviewer на обновлённый набор правок.
4. Если после **второго** ревью существенных проблем нет — приемка.
5. Если после второго ревью остаются findings, требующие правок — останови процесс и явно передай управление человеку (в плановом режиме см. правила по `todos` в [workflow-plan.md](workflow-plan.md)).

### Эскалация

При отклонении от процесса, нехватке данных или затруднениях **останови** процесс и сообщи пользователю.

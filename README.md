# ai-bro

![ai-bro](./assets/banner.png)

Когда мы постигаем что-то новое, всегда проще и удобнее это делать не в одиночку, с кем-то вместе, со своим бро... AI bro!

Работает со всеми популярными harness и оболочками! 

## Почему он, а не ...

AI bro - это набор скиллов, которые делают вашу работу с агентами гораздо удобнее, проще и эффективнее.

Он **НЕ навязывает** вам своё видение правильных паттернов и методологий, а **адаптируется** под ваши реалии и **помогает** работать в них.

## Установка

Простой и привычный для всех путь:
```bash
npx skills add irpsv/ai-bro -g
```

Либо можете попросить своего агента:
```
Установи скиллы из репозитория https://github.com/irpsv/ai-bro
```

## Механика работы

Каждую задачу можно разбить на вполне конкретные и понятные этапы:
1. **Знакомимся** - понимаем, как устроен набор, и при необходимости заводим memory bank.
2. **Думаем** - проясняем мысль, проверяем идею, выбираем подход.
3. **Дебажим** - собираем факты, проверяем гипотезы, находим основную причину.
4. **Планируем** - собираем однозначный план без кода.
5. **Делаем** - непосредственно разрабатываем то, что напланировали.
6. **Помним** - фиксируем устойчивые знания о проекте в memory bank.

И на каждом из этих этапов вам поможет ваш бро.

### Знакомимся

Когда вы только подключили набор или заходите в новый репозиторий, скажите `/bro-on-board`, и ваш бро коротко покажет карту скиллов и типовой флоу, а затем предложит инициализировать memory bank проекта.

Внутри скилла `bro-on-board` заложены два режима:
- **Tour** — знакомство с командами и этапами работы.
- **Init memory bank** — создание канонической структуры банка и первичное заполнение контекста проекта.

Если банк уже есть, скилл его не перезаписывает и направляет к `/bro-remember-it` для пополнения.

Этот скилл сам автоматически не подключается, поэтому его лучше вызывать явно с помощью slash команды `/bro-on-board`.

Пример вызова:
```text
/bro-on-board расскажи как пользоваться и предложи завести memory bank
```

### Думаем

Когда нам нужно что-то поштормить, или у нас есть проблема, но нет решения, просто скажи `/bro-think-with-me`, и ваш друган придёт на помощь и направит вас на путь истинный.

Внутри скилла `bro-think-with-me` заложены правила ведения удобного и комфортного диалога, а также поддержка разных вариантов общения:
- **Прояснение** — мысль ещё сырая, вопрос не сформулирован или в запросе смешаны несколько тем.
- **Проверка идеи** — есть идея, подход или вывод; нужно проверить логику, допущения и слабые места.
- **Поиск решения** — проблема уже понятна, но неясно, как её лучше решить или обойти.
- **Выбор подхода** — нужно что-то добавить, изменить или спроектировать; выбираем, как это сделать.

Скилл может подключаться автоматически в зависимости от текущего диалога и работает в режиме READONLY, НО для надёжности и на длинных сессиях вы можете явно вызывать его с помощью slash команды `/bro-think-with-me`.

Пример вызова:
```text
/bro-think-with-me мне нужно понять как лучше организовать удобный процесс для работы пользователя по сценарию авторизации
```

### Дебажим

Когда что-то сломалось, ведёт себя нестабильно или регресснуло после изменений, просто скажи `/bro-debug`, и ваш бро поможет спокойно докопаться до причины, не прыгая сразу к фиксу.

Внутри скилла `bro-debug` заложен аккуратный READONLY-процесс расследования с упором на доказательства:
- **Воспроизведение** — уточняем, как именно проявляется проблема и при каких условиях.
- **Сбор улик** — поднимаем код, логи, тесты и другие наблюдения, которые помогают понять происходящее.
- **Проверка гипотез** — отделяем догадки от фактов и отбрасываем то, что не подтверждается.
- **Подготовка к следующему шагу** — собираем выводы так, чтобы по ним уже можно было планировать фикс.

Скилл может подключаться автоматически в зависимости от текущего диалога и работает строго в режиме READONLY, НО для надёжности и на длинных сессиях вы можете явно вызывать его с помощью slash команды `/bro-debug`.

Пример вызова:
```text
/bro-debug после последнего деплоя периодически падает создание заказа, помоги найти причину
```

### Планируем

Когда задача уже более-менее понятна и пора превращать её в чёткий план действий, просто скажи `/bro-give-me-plan`, и ваш бро поможет собрать чёткий и понятный план, благодаря которому будет ясно, что именно нужно реализовать.

Внутри скилла `bro-give-me-plan` заложены разные форматы планирования под разные масштабы задачи:
- **План в чате** — если задача небольшая, результат понятен и отдельный артефакт не нужен.
- **Быстрый план** — если шагов уже много и такой план неудобно обсуждать или ревьюить прямо в чате, создаётся отдельный файл плана.
- **Многоэтапный план** — если задача крупная, затрагивает несколько модулей или её нужно вести поэтапно, сначала создаётся основной план с описанием задачи и этапов, а затем для каждого этапа — отдельный план, готовый к реализации.

Этот скилл сам автоматически не подключается, поэтому его лучше вызывать явно с помощью slash команды `/bro-give-me-plan`. Перед составлением плана он собирает недостающий контекст, предлагает подходящий формат и просит подтвердить выбор.

Пример вызова:
```text
/bro-give-me-plan нужно составить план для добавления rate limiting на публичный API
```

### Память проекта

Когда нужно пополнить постоянный слой знаний о проекте — контекст, фичи, типовые процедуры — скажи `/bro-remember-it`, и ваш бро аккуратно добавит знания в существующий memory bank.

Скилл `bro-remember-it` работает только с уже инициализированным банком: классифицирует знание и записывает его в нужный раздел без дублей. Если банка ещё нет — сначала вызови `/bro-on-board`.

Скилл **не** создаёт артефакты планирования в `plans/` — для планов используй `/bro-give-me-plan`.

Этот скилл сам автоматически не подключается, поэтому его лучше вызывать явно с помощью slash команды `/bro-remember-it`.

Пример вызова:
```text
/bro-remember-it добавь в memory bank описание фичи rate limiting
```

### Делаем

Когда уже пора переходить от разговоров и планов к реальной разработке, просто скажи `/bro-do-it`, и ваш бро возьмёт задачу в работу и доведёт её до результата через реализацию и ревью.

Внутри скилла `bro-do-it` заложен понятный процесс разработки: сначала задача берётся в работу и реализуется, потом результат обязательно проходит ревью, а если в нём находятся серьёзные замечания, они исправляются и проверяются повторно. То есть скилл не просто "пишет код", а организует весь цикл до внятного и проверенного результата.

Этот скилл сам автоматически не подключается, поэтому его нужно вызывать явно с помощью slash команды `/bro-do-it`. Он особенно полезен, когда у вас уже есть готовый план, либо задача и так достаточно чётко сформулирована.

Пример вызова с планом:
```text
/bro-do-it выполни план из файла plans/add-rate-limit.md
```

Пример вызова простой правки без артефактов:
```text
/bro-do-it поправь текст ошибки в форме входа и добавь тест на этот сценарий
```

### Делаем без плана

Когда идея уже прояснена и хочется не составлять план, а сразу получить работающий результат, скажите `/bro-vibe-code`, и ваш бро кратко зафиксирует, что изменится для пользователя, а затем автономно перейдёт к реализации, проверке и ревью.

Внутри скилла `bro-vibe-code` заложен режим высокой самостоятельности:
- перед стартом он даёт короткую сводку будущего пользовательского результата без технических деталей;
- не просит подтверждать внутренний план и не эскалирует технические решения;
- сам выбирает подход по коду проекта, реализует изменения, проверяет их и прогоняет ревью;
- останавливается только для действительно важных решений: пользовательский результат, безопасность, данные, доступы или необратимые внешние эффекты.

Этот скилл сам автоматически не подключается, поэтому его лучше вызывать явно с помощью slash команды `/bro-vibe-code`. Он особенно полезен после обсуждения, когда уже понятно, что должно измениться, но не хочется оформлять отдельный план.

Пример вызова:
```text
/bro-vibe-code сделай выбранный вариант авторизации из нашего обсуждения
```
